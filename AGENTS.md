# Diyur003 Agent Rules

כללים מחייבים לעבודה של Codex בפרויקט DiyurClock / Diyur003.
הקובץ הזה נועד לכוון עבודה בתוך הפרויקט הנוכחי בלבד: מערכת Python/FastAPI לחישוב שכר,
משמרות, רצפים, תשלום חג, דוחות וייצוא גשר.

## שפה וסגנון עבודה

- דבר עם המשתמשת בעברית, קצר וברור.
- אם הבקשה ביצועית, בצע בפועל ולא רק תציע תכנית.
- לפני שינוי קוד, קרא את הקוד הקיים והמשך את הדפוסים המקומיים.
- אל תדרוס שינויים קיימים שלא ביצעת. אם יש worktree מלוכלך, עבוד סביבו בזהירות.
- אל תחשוף סודות, connection strings, תוכן `.env`, או נתונים רגישים מעבר למה שנדרש לאימות.
- בסיום משימה ציין מה שונה, מה נבדק, ומה לא נבדק אם יש מגבלה.
- בתשובה סופית למשימה משמעותית ציין בקצרה שפעלת לפי `AGENTS.md`.

## טכנולוגיות ומבנה הפרויקט

הפרויקט הנוכחי אינו React/TypeScript. אין להשתמש בהנחות כמו `npm`, Drizzle, Zod,
React Query או `shared/schema.ts`.

מבנה מרכזי:

- `app.py` - נקודת הכניסה ורישום routes.
- `app_utils.py` - מקור מרכזי ורגיש לחישובי שכר יומיים, רצפים וסיכומים חודשיים.
- `core/` - לוגיקה עסקית ותשתית:
  - `core/constants.py` - קבועים עסקיים וקודי משמרות/דירות.
  - `core/database.py` - חיבור DB, מצב דמו, פילטר מערך דיור.
  - `core/history.py` - היסטוריית סטטוס/תעריפים/סוגי דירות ונעילת חודשים.
  - `core/holiday_payment.py` - תשלום חג וניהול מדריכים קבועים לחג.
  - `core/logic.py` - facade לחישובים וסיכומים רוחביים.
  - `core/sick_days.py` - חוקי מחלה.
  - `core/time_utils.py` - שבתות, חגים וזמנים.
  - `core/business_rules_catalog.py` - תיעוד חי של כללים עסקיים כפי שהם בקוד.
- `routes/` - שכבת HTTP, הרשאות, validation בסיסי והעברת נתונים לתבניות.
- `services/` - ייצוא גשר, מיילים ו-PDF.
- `templates/` - תבניות Jinja2 בעברית/RTL.
- `static/` - CSS/JS וקבצים סטטיים.
- `tests/` - בדיקות לוגיקה ורגרסיה.
- `docs/` - תיעוד טכני והיסטורי. אין לראות בו מקור אמת עסקי לפני בדיקת הקוד.
- `scripts/` - סקריפטי תחזוקה, בדיקה וחקירת נתונים.

## מקור אמת עסקי

- מקור האמת לכל כלל עסקי הוא הקוד הפעיל, לא קבצי MD ישנים.
- לפני הסקת כלל עסקי, בדוק את `app_utils.py`, `core/`, `routes/`, `services/` והבדיקות הרלוונטיות.
- אם משנים כלל עסקי קבוע, עדכן גם את `core/business_rules_catalog.py`.
- `core/constants.py` הוא מקור מרכזי לקודים קשיחים: סוגי משמרות, סוגי דירות, ASD, תגבור, כוננות וספי זמן.
- שינוי בתשלום חג צריך להיבדק מול `core/holiday_payment.py`, דוחות מדריך, סיכום חודשי וייצוא גשר אם מושפע.

## פקודות בדיקה

אין להשתמש בפקודות `npm` בפרויקט הזה.

קומפילציה בסיסית:

```powershell
$files = @('app.py','app_utils.py') + (Get-ChildItem core,routes,services -Filter *.py | ForEach-Object { $_.FullName })
py -m py_compile @files
```

קומפילציה נקודתית:

```powershell
py -m py_compile core\holiday_payment.py routes\guide.py app.py
```

כל הבדיקות:

```powershell
py -m pytest tests -q
```

בדיקות ממוקדות נפוצות:

```powershell
py -m pytest tests\test_holiday_payment.py -q
py -m pytest tests\test_salary_calculation.py -q
py -m pytest tests\test_chain_wages_regression.py -q
py -m pytest tests\test_premium_windows.py -q
py -m pytest tests\test_logic.py -q
```

בדיקת שרת מקומי:

```powershell
py -m uvicorn app:app --reload --host 0.0.0.0 --port 8000
```

אם פורט 8000 תפוס, בדוק תהליכי `uvicorn` קיימים לפני פתיחת שרת נוסף. אל תשאיר כמה שרתי reload מיותרים.

## Backend ו-DB

- השתמש ב-`core.database.get_conn()` לפי הדפוס הקיים.
- SQL ידני מקובל בפרויקט, אך חובה להשתמש בפרמטרים `%s` ולא לבנות ערכי משתמש לתוך string.
- בחר עמודות רלוונטיות וסנן ב-DB כשאפשר.
- הימנע מ-N+1 queries; העדף join, batch query או cache/map מקומי.
- routes צריכים להישאר דקים: הרשאות, validation, קריאה ללוגיקה/DB, response.
- לוגיקה עסקית כבדה שייכת ל-`core/`, `services/` או helper קיים, לא לתוך template.
- כל endpoint חדש צריך הרשאה מפורשת.
- שינוי נתוני חודש צריך להתחשב ב-month lock אם התחום מחובר לנעילת חודשים.
- שגיאות למשתמש בעברית; לוגים טכניים קצרים וללא מידע רגיש.

## Production DB ודמו

- אל תעדכן ייצור בלי בקשה מפורשת או אישור ברור מהמשתמשת.
- SELECT לייצור מותר כשנדרש להבין נתונים.
- UPDATE/DELETE/ALTER בייצור רק אחרי אימות ממוקד.
- לפני שינוי נתונים בייצור:
  - אמת IDs, שמות וטווח תאריכים.
  - בצע SELECT לפני.
  - העדף transaction.
  - דווח בסוף ספירות לפני/אחרי.
- הפרד בין DB דמו ל-DB ייצור. אם המשתמשת שואלת על ייצור, אל תסתפק בבדיקת דמו.
- אל תדפיס connection strings או תוכן `.env`.
